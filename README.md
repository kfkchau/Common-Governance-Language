# Common Governance Language (CGL)

The **Common Governance Language (CGL)** is a shared descriptive grammar for governed reality.

Its purpose is to allow policy, operations, compliance, and technical teams to describe the same governed system in compatible terms. Within the wider **Open Governance Standard (OGS)** stack, CGL functions as the **language layer**. Where OMGS defines how rule systems change, CGL defines how governed reality is described. The Task Boundary Framework uses that descriptive logic to define governable work, and PWC uses it in runtime orchestration and governed execution contexts.

## What CGL is

CGL is **not** a governance standard in the same sense as OMGS. It is a **modular language layer** made up of interoperable component grammars for describing the elements, structures, and relationships through which governance operates.

CGL exists to reduce translation loss between different institutional and implementation layers. In many real systems, law, policy, procedure, workflow design, service operations, compliance functions, and technical implementation all refer to the same governed reality, but in incompatible vocabularies. CGL is intended to provide a common descriptive layer across those contexts.

## Naming and lineage

CGL is the **current governance-facing identity** of work developed through the earlier **Logic Atlas** lineage.

The Logic Atlas materials remain the development history for this language layer. They contain much of the underlying schema work, integration logic, and modular decomposition on which CGL is based. What has changed is not the existence of that developmental lineage, but the foreground framing, naming, and public-facing governance purpose.

In practical terms:

- **CGL** is the current front door and preferred current name
- **Logic Atlas** remains the historical lineage and development history
- the underlying conceptual work is continuous across both

## What CGL covers

CGL is organised as a set of modular component grammars for describing governed systems. These include:

- **Activity grammar** — how governed activities are structured
- **Action grammar** — how actions are classified and tagged
- **Resource grammar** — how resources are described and differentiated
- **Enforcement grammar** — how interventions vary by mechanism and scope
- **Measurement grammar** — how governed entities and effects are quantified
- **Relationship / architecture grammar** — how entities and flows are arranged
- **Geometry / composition grammar** — how things are positioned, nested, and composed
- **Connection logic** — how activities and structures are linked through conditions and logic

Together, these component grammars allow CGL to function as a shared language for governed reality.

## Core framing

CGL is a shared descriptive grammar for governed reality. Component grammars are developed across the major descriptive domains through which governance is represented.

**The grammar of governed activity**  
Governed reality can be described through grammars for:

- activities
- actions
- resources

**The grammar of governed structure**  
Governed reality can also be described through grammars for:

- relationship architecture
- spatial / relational geometry
- composition / part logic

**The grammar of governed condition, effect, and control**  
Governed systems further require grammars for:

- connection logic
- enforcement logic
- measurement logic

Together, these grammars allow CGL to function as a common language layer across governance, operations, compliance, and technical implementation.

## Interoperability role

CGL is intended to support interoperability in two directions:

**Vertical interoperability**  
Across layers such as law, policy, procedure, workflow, and executable control.

**Horizontal interoperability**  
Across sectors and governance domains such as social services, health, finance, procurement, foreign affairs, and AI-enabled systems.

Its function is to help different parts of a governed system describe the same reality coherently enough to coordinate, implement, audit, and redesign it.

## Relationship to OGS

CGL is part of the broader **Open Governance Standard (OGS)** stack.

Within that stack:

- **OMGS** defines how rule systems change over time
- **CGL** defines how governed reality is described
- the **Task Boundary Framework** uses that descriptive logic to define governable work
- **PWC** uses it in runtime orchestration and governed execution contexts

CGL therefore acts as the shared language layer that allows the different parts of the stack to speak about the same system coherently.

## Relationship to Logic Atlas materials

Where older documents, repositories, or notes refer to **Logic Atlas**, they should generally be understood as part of the same underlying language-development effort now being expressed under the **CGL** identity.

The preferred current naming is:

- **CGL** for the current governance-facing language layer
- **Logic Atlas** for historical lineage, prior framing, and development history
